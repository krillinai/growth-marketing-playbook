# 增长基础设施

> 增长基础设施把重复衡量、投放、生命周期、素材与决策工作转化成可复用系统。只有产品共享稳定问题且本地团队保留必要语境时，集中化才有价值。

**[English](README.md)｜[简体中文](README.zh.md)**

## 核心问题

- 哪项重复能力具有稳定输入与输出？
- 什么必须保持产品或市场特定？
- 自动化、数据访问、风险与审计如何控制？

## 跨公司模式

横向能力只有在连接具体增长环节和客户价值时才有意义。成熟做法会同时设计流程、责任、数据、反馈与风险边界。

## 公司证据

| 公司 | 证据与方法 |
| --- | --- |
| 字节跳动 | LTV 模型、自动投放、素材系统、归因、风控、共享 SDK 与产品矩阵协调。 |
| Facebook / Meta | 埋点、实验、翻译、引导与扩张系统。 |
| Ramp | 增长工程、数据与 GTM 连接，以及共享 AI Agent 语境。 |

## 关键指标

- 内部团队采用率
- 周期缩短
- 决策准确性与人工覆盖率
- 数据新鲜度与事故率
- 增量下游价值

## 可以迁移的方法

1. 建设平台前先梳理产品间重复工作。
2. 集中稳定能力，而不是所有决策。
3. 定义负责人、权限、日志、人工覆盖与停止条件。
4. 同时衡量内部采用与客户结果。

## 不应该直接复制的部分

- 不要自动化不稳定或不可审计的流程。
- 不要复制为自身不具备的规模、监管或产品矩阵建设的基础设施。

## 证据与局限

- 证据主要来自参与者访谈和公开材料，不是完整内部因果评估。
- 公司规模、品类、监管、数据与人才条件限制直接迁移。
- 发布方增长、收入、估值与客户数主张不用于证明单项方法。
