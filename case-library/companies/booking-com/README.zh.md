# Booking.com

> 一个围绕去中心化产品决策、自助式基础设施、统一方法、并行实验与组织学习展开的实验体系案例。

**[English](README.md)｜[简体中文](README.zh.md)**

## 增长背景

Booking.com 把在线实验建设成广义产品开发能力，而不是集中式分析服务。Lukas Vermeer 对其职责的描述覆盖基础设施与工具、方法与指标、培训以及文化。

这一案例展示组织如何在保留统一质量控制的同时提高决策吞吐量。公开演讲不能证明实验体系的准确增量业务影响，也不代表 Booking.com 当前全部实践。

## 去中心化实验体系

```text
产品团队拥有客户问题与假设
-> 共享平台管理分流、曝光与分析
-> 统一方法和指标保护可比性
-> 培训支持本地判断
-> 证据改变产品决策
-> 文档沉淀为可复用的组织学习
```

去中心化不等于没有标准。体系需要区分可复用的统一控制，以及必须保留在产品场景附近的决策。

| 层级 | 可复用能力 | 需要避免的故障 |
| --- | --- | --- |
| 基础设施 | 分流、曝光、事件数据、分析与诊断 | 每个团队建设互不兼容的 Pipeline |
| 方法 | 设计标准、不确定性、质量检查与结果解释 | 自助实验产生自信但无效的结论 |
| 指标 | 统一定义、结果指标、诊断指标与护栏 | 团队在看到结果后挑选有利指标 |
| 培训 | 产品与技术角色的实操能力 | 知识集中在少数专家手中 |
| 文化 | 允许挑战直觉并停止薄弱想法 | 实验变成确认既有观点的表演 |

## 并行实验

较高的实验吞吐量意味着部分测试需要重叠。全面互斥虽然规避了理论上的交互，却会带来确定的容量成本。

| 情况 | 处理方式 |
| --- | --- |
| 机制和界面相互独立 | 允许重叠，并保留曝光记录 |
| 共享页面或稀缺资源 | 评估冲突并监控重要组合 |
| Treatment 会直接使彼此失效 | 使用互斥或组合设计 |
| 出现异常异质性结果 | 检查并行曝光和合理的交互机制 |
| 高风险或不可逆后果 | 使用更强隔离与审核 |

交互分析应从合理机制和决策影响出发。搜索所有 Treatment 组合也会产生多重检验和解释问题。

## 增长基础设施经验

- 让标准且可信的工作流可以自助使用。
- 集中管理可复用的平台可靠性和指标治理，而不是集中每项产品决策。
- 在重复工作和故障模式出现后再建设自动化。
- 保存分流、曝光、分析和决策文档，使结果可以被审核。
- 衡量证据是否改变决策，而不只计算启动了多少实验。
- 把基础设施、方法、培训和文化作为一个系统共同发展。

## 迁移边界

- 小型组织可能没有足够流量或实验量支撑 Booking.com 规模的基础设施。
- 去中心化要求本地团队具备足够的统计和产品判断能力。
- Marketplace 干扰、重复曝光、季节性与共享库存可能需要超出用户级随机化的设计。
- 高频实验不能弥补战略、客户理解或产品价值不足。
- 平台投入应由当前约束决定，而不是照搬另一家公司的规模。

## 证据与局限

- 案例基于公开会议演讲和访谈，不是内部平台文档或独立因果评估。
- 不同日期的演讲可能描述体系的不同阶段。
- 不主张任何准确的增量收入、转化、留存或创新效应。
- Booking.com 的结果还来自产品、Marketplace、品牌、运营、战略、资本、时机与团队。
